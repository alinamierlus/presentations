Explaining Firefox OS

All in all we gave thousands of demos and hundreds of press interviews about the OS.

The VW Beetle of Smartphones

Sometimes it was tough to describe what Firefox OS is and how it differentiates itself from the already existing mobile platforms. The most basic explanation for non-technical journalists I could come up with was that Firefox OS is the VW Beetle of Smartphones. The original Beetle, that is.

The VW Beetle of Smartphones?

  • Affordable by everyone
  • Built on sturdy technology
  • Lots of interchangeable parts - adding and upgrading is easy
  • Built on a trusted engine that doesn't boil over
  • Making mobility and independence affordable
When the Beetle got released, its main goal was to be affordable by everyone and almost impossible to break. Its engineering and air-cooled engine made it a very sturdy means of transportation for people who beforehand couldn't afford a car.

Replacing feature phones

Comparing Firefox OS with Android or iOS head-on is therefore a pointless idea. The aim of Firefox OS is to replace feature phones in emerging markets. True to the main goal of Mozilla to keep the web open and affordable to everyone. It can not be that a means of mass communication and information is only for the rich in certain markets. The web is a voice for a lot of people who can not speak otherwise as their mass media is controlled. Therefore mobile web access needs to be affordable and easy for everyone.

HTML5 through and through…

Photo by Alden Jewell
The one and only technology used in Firefox OS is HTML5. The interface of the operating system is written in it, all apps are and the access to the hardware is defined in JavaScript APIs. This means that we can re-use a lot of the already existing code on the web and the system is very easy to upgrade and extend. Apps tend to be smaller and simpler, but also more cost-efficient to use. This is also something that is very important in emerging markets.

But HTML5 can't do…

Almost everyone we talked to at Mobile World Congress had preconceptions of HTML5 being unable to do certain things that native apps can. This ranged from 3D gaming up to working offline. Most of these assumptions are wrong, and the reason is how we measure the usefulness of HTML5.

False measures

A lot of what HTML5 is accused of failing at is actually the fault of native platforms not allowing browsers to access function-critical hardware, or plain wrong implementations in stock browsers. This issue is exacerbated as stock browsers only update when the OS updates. That way older hardware is doomed to display HTML5 sub-par.

Give me that hardware!

https://wiki.mozilla.org/WebAPI

This was one of the main reasons why Mozilla came up with Firefox OS - back then called Boot to Gecko. We wanted to find a way to access the hardware that makes mobile devices interesting from the web stack. And that we did. Using open standards - the Web APIs - defined and shared across the market we allow JavaScript developer to access the hardware in mobile devices directly without having to convert to Java or Objective-C.

Open Web Apps?

https://developer.mozilla.org/en-US/docs/Apps

Of course from a security point of view there is no way we can allow any unknown server and code to access all the hardware on the device. That is why Firefox OS runs a packaged App format called Open Web Apps.

Marketplace ✔

Firefox Marketplace

Firefox OS has a marketplace for apps - much like any other platform out there. The Marketplace itself is an HTML5 app based on APIs for installing, de-installing and billing for apps. This allows partners to also build their own white-labeled marketplaces.

Find app by content ✔

The really cool differentiator though is that as the apps are HTML5, they can be indexed and searched. That way Firefox OS offers a functionality to search apps by content type. This means that if you enter the name of a band, you get apps with their music, their lyrics, videos or just information about them offered to you. These are mobile HTML5 apps which means you can try them out and install them in a second step if you like them. A try before you buy so to say. With Firefox OS we made apps as easy to discover and consume as the web is right now.

Defined by manifest

        {
          "name": "My App",
          "description": "My elevator pitch goes here",
          "launch_path": "/",
          "icons": { "128": "/img/icon-128.png" },
          "developer": {
            "name": "Your name or organization",
            "url": "http://your-homepage-here.org"
          }
        }

App manifest

You define a manifest file to describe your app and tell the workd what icons to use for it, what access it needs and which servers it can be installed from.

Three levels of access…

  • Hosted apps - stored on your server, easy to upgrade, limited access.
  • Privileged apps - reviewed by the App store, uses a Content Security Policy, hosted on trusted server
  • Certified apps - part of the OS, only by Mozilla and partners

App permissions

Open Web Apps come in three different versions that allow for more or less access to the hardware. For third party developers, hosted and privileged apps are the ones you can build. Whilst hosted apps are allowed to do all the things HTML5 apps on the web can do, privileged apps can dig deeper into the platform.

Using WebAPIs to make the web layer more capable

The Web APIs available to hosted apps can already get you very far. A lot of them are already available across browsers and on Desktop, like the Battery API. Interesting are also Push Notifications and Vibration for interaction with the user.

Install apps from the web

        var installapp = navigator.mozApps.install(manifestURL);
        installapp.onsuccess = function(data) {
          // App is installed
        };
        installapp.onerror = function() {
         // App wasn't installed, info is in 
         // installapp.error.name
        };
A very simple way to make your app installable on a Firefox OS device is to use the install() method. This means you don't have to wait for a marketplace review and listing but can turn any web site into an app by people surfing to it.

Battery API

          var b = navigator.battery;
          if (b) {
            var level=Math.round(battery.level * 100) + "%",
                charging=(battery.charging) ? "" : "not ",
                chargeTime=parseInt(battery.chargingTime / 60, 10),
                dischargeTime=parseInt(battery.dischargingTime/60,10);
            b.addEventListener("levelchange", show);
            b.addEventListener("chargingchange", show);
            b.addEventListener("chargingtimechange", show);
            b.addEventListener("dischargingtimechange", show);
          }
Using the battery API you can notify the user when the energy level is low and offer for example to switch to a simpler interface or to send a warning that a long download might not work out.

Using WebAPIs to make the web layer more capable

Privileged apps can do more than hosted ones as the server is certified and known. For example you can call other (CORS enabled) servers with XHR or store and read content on the device outside of indexedDB.

Contacts API

          var contact = new mozContact();
          contact.init({name: "Christian"});
          var request = navigator.mozContacts.save(contact);
          request.onsuccess = function() {
            // contact generated
          };
          request.onerror = function() {
            // contact generation failed
          };
For a hosted, privileged app it is simple to create a new contact. That enables you to sync address books across services, for example.

Using WebAPIs to make the web layer more capable

Certified apps are applications that are part of the operating system and are right now only built by Mozilla and partners. They have full access to the phone's capabilities.

Why not all for hosted apps?

It can appear annoying that not all HTML5 apps can access everything in the phone but the sad truth is that it is too easy to abuse a system that would be this open. Therefore we need to have a security review and capability framework in place. This doesn't mean though that you can not access the hardware at all - it means that you can ask the end user to do it on your behalf.

Web Activities

  • configure
  • costcontrol
  • dial
  • open
  • pick
  • record
  • save-bookmark
  • share
  • view
  • new, f.e type: “websms/sms” or “webcontacts/contact”

https://wiki.mozilla.org/WebAPI/WebActivities

Web Activities is a standards proposal to trigger a user interaction that is part of the operating system. Instead of using the functionality of the OS you use it with a user-initiated action. That way you don't have the security issue you'd have by accessing the functionality on behalf of the users and without notifying them. This is a concept from Android where it is called intents. Chrome until recently also had Web Intents but pulled them a short while ago because of UX issues on the Desktop.

Giving control to the user

When initiating a Web Activity, the end user gets prompted to allow it. For example a pick activity would ask them to get images from the pictures or wallpaper folder or take a new photo. This is the beauty of activities - as a developer I don't care where content comes from or where it goes. Instead I leave that up to the user. Much like a right-click on any file on your hard-drive allows you to open it in your program of choice.

Sending a number to the phone

        var call = new MozActivity({
          name: "dial",
          data: {
            number: "+1804100100"
          }
        });
This is all that is needed to send a phone number to the dialer. All the user needs to do is hit the receiver button to call it.

Get an image from the phone (1/2)

        var getphoto = new MozActivity({
          name: "pick",
          data: {
            type: ["image/png", "image/jpg", "image/jpeg"]
          }
        });
Getting an image is done by invoking the pick activity and defining the type of files you expect.

Get an image from the phone (2/2)

        getphoto.onsuccess = function () {
          var img = document.createElement("img");
          if (this.result.blob.type.indexOf("image") != -1) {
            img.src = window.URL.createObjectURL(this.result.blob);
          }
        };
        getphoto.onerror = function () { // error
        };
Once invoked, you have success and error event handlers to retrieve the image as a blob.

You want some of that?

I hope you now have some idea of how amazing an offer Firefox OS is for developers and how it gives HTML5 the platform it deserves. Following is now information where you can go to learn more.

Developer Hub

Firefox OS Developer Hub

The Firefox OS Developer Hub is where we put all the info you need. You find design guidelines, demo apps and tools to validate your app manifest and get started with the submission process to the Marketplace.

Developer tools

A lot of people ask about IDEs and development environments. There are no fixes ones. As Firefox OS uses the same engine as Firefox on Desktop you can use the Developer Tools in the browser to get 90% of the way to your own app. You can use the Responsive Design Mode to see your app in 320 x 480 resolution and you can analyse the CSS, JavaScript and net requests with the in-build inspectors.

Simulator in the browser

Firefox OS Simulator 1.0 is here!

The Firefox Simulator is an add-on for the browser that gives you a full Firefox OS instance in an own window, complete with debugging console and the opportunity to install local apps without needing a server.

Firefox OS Boilerplate

Boilerplate App on GitHub

My colleague Robert Nyman maintains a great demo app on GitHub that acts as a perfect starting point for your app. It has a responsive interface and demo buttons to invoke all the web activities. All you need to do is to remove the ones you don't need and add your functionality.

Phones for developers

Geeksphone.com

If you want to get your hands on a Firefox OS device before it comes to the market, there is a company called Geeksphone in Spain that offers affordable developer mobiles with the same specifications. You can pre-order them now.

Activities/Hosted Apps on Android

Even if you don't care for Firefox OS, using Firefox on Android brings a lot of the functionality of the operating system to Android phones. This makes it possible to even bring very outdated mobiles up to date on HTML5 support.

The web ride is far from over!

All in all we can be assured that the web is far from dead and replaced by walled gardens. If anything, I am sure that a lot of predictions that everything just works in one environment and as developers it makes most sense to concentrate on that will become very much outdated this year. The web is an existing, world-wide distribution platform and reaches millions of people already with many more to come. Firefox OS is one step to bring web mobile connectivity to those who can not afford the other platforms. This is exciting and very much in the spirit of the web. And nothing stops you from packaging your HTML5 apps as native. The other way around doesn't work.
